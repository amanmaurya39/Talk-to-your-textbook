# ADR-001: ChromaDB as Vector Store

**Date:** July 2026  
**Status:** Accepted  

---

## Context

The Talk To Your Textbook app needs a vector store to:
- Store embeddings of PDF chunks (400-token pieces)
- Perform fast similarity search when a user asks a question
- Persist data across app restarts (so we don't re-embed every time)
- Run locally during development without any cloud setup

I evaluated 4 options: ChromaDB, Pinecone, Qdrant, and pgvector.

---

## Decision

**Use ChromaDB with PersistentClient.**

---

## Reasons

- Runs 100% locally — no account, no API key, no internet needed
- PersistentClient saves to disk — embeddings survive app restarts
- Zero configuration — works in 3 lines of Python
- Free forever for local use — no cost surprises
- Good enough performance for documents up to ~500 pages

---

## Consequences

**Positive:**
- Setup time: 5 minutes vs hours for cloud options
- No ongoing cost
- Works offline
- Easy to reset (just delete the chroma_db/ folder)

**Negative:**
- Not scalable beyond 1 user/session (no multi-tenant support)
- No built-in cloud sync (but not needed for this internship project)
- Slower than Pinecone at very large scale (10M+ vectors)

---

## Alternatives Considered

| Option | Why Rejected |
|---|---|
| **Pinecone** | Requires account + API key, free tier has limits, overkill for local dev |
| **Qdrant** | Good option but more complex setup, Docker required |
| **pgvector** | Requires PostgreSQL setup, too much overhead for a 5-week project |

---

## Review (3rd Year)

In 3rd year, if this becomes a multi-user SaaS product, migrate to 
**Qdrant Cloud** or **Pinecone** for proper multi-tenant support.
