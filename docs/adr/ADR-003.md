# ADR-003: Hybrid Retrieval over Dense-Only

**Date:** July 2026  
**Status:** Accepted  

---

## Context

The Talk To Your Textbook app requires a highly accurate document retrieval mechanism to:
- Find the most relevant page chunks from a textbook containing answers to user questions
- Handle both semantic/conceptual queries (e.g., "how optimization works") and keyword-specific queries (e.g., "ReLU" or "Adam")
- Keep execution lightweight and local without external database dependencies

I evaluated dense-only retrieval, sparse-only retrieval (BM25), and hybrid retrieval combining both.

---

## Decision

**Use Hybrid Retrieval (ChromaDB dense similarity with Google embeddings + BM25 keyword re-ranking).**

---

## Reasons

- **Best of Both Worlds:** Dense retrieval (using Google `gemini-embedding-001` or `text-embedding-004`) matches concepts and synonyms but can miss exact names, symbols, or code terms. Sparse retrieval (BM25) ensures exact terms are matched directly, ensuring the right pages are prioritized.
- **Accurate Citations:** By combining dense and sparse search scores, the retrieved context chunks contain the exact terms queried, reducing retrieval error rate and producing correct page citations.
- **Lightweight Local Setup:** Both ChromaDB and the `rank_bm25` python package run entirely in memory/local disk, requiring no cloud database infrastructure.

---

## Consequences

**Positive:**
- Significant improvement in retrieval accuracy, especially for queries asking about specific formulas, parameters, or terms
- Keeps the system self-contained and free of external SaaS database charges
- Fast response: BM25 re-ranking is performed on a pre-filtered list of candidate chunks, keeping latency low

**Negative:**
- Increased code complexity: requires computing dense embeddings, performing a vector search, running a BM25 tokenized match, and merging the scores
- Tuning the combination weights (currently 70% dense + 30% BM25) is empirical and depends on document style

---

## Alternatives Considered

| Option | Why Rejected |
|---|---|
| **Dense-Only Retrieval** | Often fails on specific, rare technical terms or acronyms, causing the model to receive incorrect page contexts. |
| **Sparse-Only (BM25)** | Cannot handle synonyms, rephrases, or general semantic queries where the exact keywords are not present in the text. |
| **Cloud-based Hybrid Retrieval (e.g. Pinecone)** | Adds unnecessary configuration, account management, internet latency, and potential pricing/billing issues. |

---

## Review (3rd Year)

If the document database grows to millions of chunks across thousands of books, migrate to a dedicated cloud search cluster supporting native hybrid indexes (like Qdrant or Pinecone) to perform retrieval at scale.
